# AI Behavior Guidelines

## Documentation Policy

- **НЕ создавать .md файлы** автоматически без явного запроса пользователя и если они не являются абсолютно необходимыми.
- **НЕ создавать документацию** типа README, SUMMARY, CHANGELOG без прямого указания
- **НЕ создавать файлы** для "документирования процесса" или "резюме работы"
- Если нужно что-то записать - использовать только комментарии в коде или краткий ответ в чате
- Делайте то, о чем вас попросили; ни больше, ни меньше.

## Code Analysis Requirements

### Обязательное изучение структуры проекта:
1. **ВСЕГДА читать** существующие файлы перед внесением изменений
2. **Детально изучать** связанные компоненты (контроллеры, сущности, репозитории, сервисы)
3. **Проверять** существующие маршруты, шаблоны и конфигурации
4. **Анализировать** зависимости между компонентами
5. **Сохранять** свой прогресс и планы в .md файлах, чтобы продолжить с того места, где вы остановились.

### Особое внимание к функционалу:
- При работе с определенной функцией - изучить ВСЕ связанные файлы
- Проверить существующие методы в контроллерах
- Изучить структуру базы данных через сущности
- Понять логику работы через сервисы
- Проверить существующие шаблоны и их структуру
- ВСЕГДА предпочитайте редактирование существующего файла созданию нового.

### Перед любыми изменениями:
1. Прочитать текущий код
2. Понять архитектуру
3. Проверить существующие паттерны
4. Следовать установленным конвенциям

## Response Style

- Отвечать кратко и по делу
- Не создавать избыточную документацию
- Фокусироваться на решении задачи
- Проверяйте функциональность по мере разработки.
- Минимальные резюме в конце работы (1-2 предложения максимум)

## File Creation Rules

- Создавать ТОЛЬКО запрошенные файлы
- НЕ создавать "полезные" файлы без запроса
- НЕ создавать документацию "на всякий случай"
- Спрашивать разрешения перед созданием любых .md файлов
- Сохранять свой прогресс и планы в .md файлах, чтобы продолжить с того места, где вы остановились.


## After completing each task
- Все тесты пройдены успешно.
- Код соответствует стандартам качества.
- Функциональность соответствует требованиям
- Интеграция с существующим кодом работает.
- Документация обновлена.

## Управление статусом задач
Поддерживайте актуальность статуса задачи:
Задача не начата : задача еще не начата
В процессе : Активно работаем над задачей.
Заблокировано : Невозможно продолжить из-за внешних факторов.
Завершено : Задача полностью выполнена и проверена.

## Отчет о ходе работ
Регулярные обновления должны включать:
Выполненные задачи : Что завершено
Текущая задача : над чем вы сейчас работаете?
Предстоящая работа : Следующие задачи в очереди
Препятствия : любые проблемы, мешающие прогрессу.
Сроки : Ожидаемые даты завершения

## Обновления документации
В процессе внедрения:
Комментарии к коду : Объясняют сложную логику и принимаемые решения.
Обновления файла README : Поддерживайте актуальность инструкций по настройке и использованию.
Архитектурные примечания : Документируйте все изменения в проекте.
Уроки, извлеченные из опыта : зафиксируйте полученные данные для будущих проектов.

## Когда следует отклоняться от плана
Допустимые отклонения
Лучшее техническое решение : найден более эффективный подход.
Упрощенная реализация : позволяет добиться того же результата гораздо проще.
Оптимизация производительности : обнаружены улучшения эффективности.
Повторное использование кода : Возможность использования существующих компонентов.

## Процесс внесения изменений
Задокументируйте предлагаемые изменения : почему и что именно изменится.
Оцените влияние : как это повлияет на другие задачи или требования?
Обновить документацию : при необходимости внести изменения в технические условия.
Коммуникация : Информируйте заинтересованные стороны о существенных изменениях.
Проверка : Убедитесь, что требования по-прежнему выполняются.

## Итеративное улучшение
Ретроспективы : Регулярный анализ того, что работает, а что нет.
Усовершенствование процесса : корректировка подхода на основе накопленного опыта.
Оценка инструментов : Рассмотрите более подходящие инструменты или методы.
Обмен знаниями : документирование полученных результатов для будущих проектов.

## Успех на уровне задачи
Функциональность : Функция работает в соответствии с заявленными характеристиками.
Качество : Код соответствует стандартам и является поддерживаемым.
Проверка : Соответствующие тесты проведены и проходят успешно.
Документация : Внедрение должным образом задокументировано.

## Успех на уровне проекта
Удовлетворенность требованиями : Все критерии приемки выполнены.
Удовлетворенность заинтересованных сторон : реализованная функция отвечает потребностям пользователей.
